# KBLens 改进建议与产品化路线

本文档整理了对 KBLens 当前状态的评审结论，以及建议的优先改进方向、优先级分层和近期实施路线。目标不是一次性堆很多功能，而是先把项目从“能跑的工具”推进到“可信、可用、可持续演进的产品雏形”。

## 1. 当前判断

### 1.1 项目是否有用

结论：有用，而且解决的是一个真实问题。

KBLens 的价值不在“生成 Markdown”本身，而在于它试图为大型代码库和文档集合建立一层可导航、可总结、可复用的知识层，帮助人和 AI 助手更快理解项目结构，而不是每次都从源码全文开始阅读。

它适合的典型场景包括：

- 大型多语言代码库的快速理解
- 遗留系统或缺少架构文档的仓库接手
- AI 编程助手需要结构化上下文的场景
- 多源项目中代码与文档的统一知识库构建

### 1.2 当前更像什么

结论：现在已经不是玩具，但还没有完全达到成熟产品状态。

它已经具备：

- 明确的核心问题定义
- 清晰的多阶段处理流水线
- 可工作的增量、并发、断点续传和分层输出
- 面向人和 agent 的潜在使用场景

它当前仍欠缺：

- 更稳定可信的增量与清理语义
- 更强的测试保护
- 更低的首次使用门槛
- 更好的检索、追溯和产品化体验

## 2. 评审结论摘要

### 2.1 方向是对的

KBLens 的核心思路是合理的：

1. 先做扫描和结构提取
2. 再做按上下文预算分批压缩
3. 再做分层摘要
4. 最终生成一个可浏览的知识库

这比“直接把整个 repo 丢给 LLM”更工程化，也更有机会做到稳定、低成本和可复用。

### 2.2 当前最需要优先解决的问题

从代码评审角度，当前最值得优先处理的不是“更多新功能”，而是结果可信度。主要问题集中在：

1. 组件边界与增量哈希边界不一致
2. 删除后的知识库产物清理不彻底
3. `include_extensions: auto` 存在漏检风险
4. 主链路测试覆盖不足，尤其是扫描、增量和清理路径

这些问题会直接影响：

- 增量更新是否可信
- 知识库是否和源码真实状态一致
- 用户是否会长期信任输出结果

## 3. 优先改进建议

整体原则：先解决“会不会错”，再解决“好不好用”，最后解决“能不能规模化协作”。

### 3.1 第一优先级：先修正确性和可信度

建议优先处理以下事项：

1. 修复组件边界与增量哈希不一致问题
2. 补全删除组件后的 package 概览和空目录清理
3. 修复 `include_extensions: auto` 的采样偏置和漏检风险
4. 为上述逻辑补关键回归测试

原因：

- 这是产品可信度的底座
- 不修好，后面的 UI、搜索、发布和协作能力都会建立在不稳定结果上

### 3.2 第二优先级：补使用体验

在结果可信之后，建议优先增强：

1. 初始化向导与配置诊断
2. 预览模式和生成成本预估
3. 结果来源追溯
4. 基础搜索能力

原因：

- 首次使用门槛决定用户会不会留下来
- 来源追溯决定用户能不能核验结果
- 搜索决定知识库是不是“真的能用”

### 3.3 第三优先级：增强产品化能力

在前两层相对稳定之后，再考虑：

1. 更完整的本地 Web UI
2. 成本与耗时面板
3. 失败分类与恢复可观测性
4. CI 集成与自动发布
5. 多仓库或多源统一管理

原因：

- 这些能力会明显提高产品上限
- 但前提是底层输出已经足够稳定、可信、可核验

## 4. P0 / P1 / P2 优先级分层

### P0：必须先做

这一层不解决，后面的功能价值都会打折。

#### P0.1 增量更新可信

目标：让“改了哪里就只重算哪里，删了哪里就同步清理哪里”成为稳定行为。

建议内容：

1. 为组件显式建模边界语义，例如 `direct` / `recursive`
2. 让扫描阶段明确标注组件边界
3. 让 `source_hash` 与组件边界保持一致
4. 统一 direct-file 组件与 `_root` 组件语义
5. 补齐组件删除后的 package 和空目录清理

验收标准：

1. 修改子组件文件，不应误触发包根 direct 组件重跑
2. 删除最后一个组件后，不应残留旧 package 页面
3. 连续两次生成的变更报告稳定且可解释

#### P0.2 自动检测不要漏内容

目标：不要因为采样上限或目录顺序导致静默漏掉某种语言或文件类型。

建议内容：

1. 改进 `include_extensions: auto`
2. 优先保证完整性，不要默认做高风险截断采样
3. 保持 `ignore_extensions` 的现有行为
4. 明确 code source 和 document source 的检测规则

验收标准：

1. 多语言仓库不会因为采样偏置漏掉扩展名
2. 输出里能清楚看到最终参与处理的扩展集合

#### P0.3 主链路回归测试

目标：把最容易回归的路径钉住。

优先测试范围：

1. 扫描边界测试
2. 增量哈希测试
3. 删除清理测试
4. auto extension 检测测试

验收标准：

1. 上述高风险场景都有自动化测试覆盖
2. 后续调整扫描和写入逻辑时，不会轻易回归

### P1：尽快补上

这一层解决“新用户能不能顺畅上手，以及结果能不能被快速验证”。

#### P1.1 初始化向导与配置诊断

目标：降低首次使用门槛。

建议内容：

1. 自动发现候选 source 目录
2. 推荐扩展名和排除规则
3. 检查 API key、依赖和输出目录
4. 生成前给出 preview

#### P1.2 结果来源追溯

目标：让用户知道“这段总结依据是什么”。

建议内容：

1. 在 component 级结果中更清晰地展示来源文件
2. 在 package 和 index 级结果中展示来源范围
3. 后续支持从页面跳到 leaf 或原始文件路径

#### P1.3 基础搜索能力

目标：让知识库真正可用，而不只是静态文档集合。

建议内容：

1. 按 package、component 搜索
2. 按 symbol、file path 搜索
3. 支持从结果直接跳转

### P2：增强型产品能力

这一层不是立刻必须做，但会决定长期产品上限。

#### P2.1 运行可观测与成本控制

建议内容：

1. token 统计
2. 分阶段耗时
3. 失败分类和重试统计
4. 快速 / 标准 / 深度模式

#### P2.2 更完整的浏览体验

建议内容：

1. 更完整的本地 Web UI
2. 分层导航和面包屑
3. 关系视图
4. 版本切换与差异比较

#### P2.3 团队协作能力

建议内容：

1. CI 集成
2. 自动发布知识库
3. 多仓库管理
4. 手工注释保留与融合

## 5. 建议的近期实施顺序

### 第一批：必须先做

建议先完成以下 4 项：

1. 哈希边界一致性修复
2. 删除清理修复
3. auto extension 检测修复
4. 对应回归测试补齐

这 4 项完成后，项目会明显从“能跑的工具”提升到“结果基本可信的工具”。

### 第二批：最小产品化体验

建议随后处理：

1. 初始化向导 / 配置诊断
2. preview 预估
3. 结果来源追溯

这一步会显著降低第一次使用成本，也会提高用户对结果的信任度。

### 第三批：让知识库更好用

建议再往后补：

1. 基础搜索
2. 更好的浏览入口
3. 成本和耗时统计

这一步会让输出从“生成结果”变成“真正可用的知识层”。

## 6. 如果只允许做 5 件事

如果当前资源非常有限，我建议优先只做下面 5 件事：

1. 修复组件哈希边界
2. 修复删除清理残留
3. 修复 auto extension 漏检
4. 补增量 / 清理 / 扫描回归测试
5. 增加初始化诊断与 preview

这 5 件事的投入产出比最高，能最快提升项目的可信度和试用体验。

## 7. 如果只允许做最小代码修改

如果目标是“最小修改但先解决最大问题”，建议优先改以下文件：

1. `src/kblens/models.py`
2. `src/kblens/scanner.py`
3. `src/kblens/writer.py`
4. `tests/` 下新增少量针对性测试文件

原因：

- `models.py` 适合补组件边界语义
- `scanner.py` 负责在组件创建时标注边界
- `writer.py` 集中了哈希、元数据和删除清理逻辑
- 测试则用于固定这次修复的关键行为

如果想更完整一点，再补：

5. `src/kblens/config.py` 或 `src/kblens/scanner.py` 中与 auto extension 检测相关的逻辑

## 8. 不建议过早优先做的事项

以下方向可以做，但不建议现在排在太前面：

1. 花哨图谱或复杂关系可视化
2. 很重的 Web UI 重构
3. 大量新增语言支持
4. 复杂的云端账户和权限体系

原因：

- 这些能力会放大产品表象
- 但无法替代“结果是否准确、可解释、可维护”这个根基

## 9. 产品化判断标准

可以用下面这句话判断 KBLens 是否开始像一个产品：

> 一个新用户不用读源码、不需要大量手改配置，就能在 10 分钟内生成一个不明显出错、可搜索、可追溯的知识库。

如果这个标准还达不到，就继续优先做 P0 和 P1。

## 10. 最终建议

最推荐的执行顺序如下：

1. P0.1 增量可信
2. P0.2 auto detection 完整性
3. P0.3 回归测试
4. P1.1 初始化诊断
5. P1.2 来源追溯
6. P1.3 基础搜索
7. P2.1 成本 / 耗时面板
8. P2.2 更完整的浏览和协作能力

整体原则不变：

1. 先解决“会不会错”
2. 再解决“好不好用”
3. 最后解决“能不能规模化协作”
